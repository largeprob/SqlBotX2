# 关键定义
> 包含 SQLBotXAgent 产品中内置的系统标准口径定义。
> 当用户提出特定关键词时，代理能够识别并理解其在业务查询、 NL2SQL 转换、数据分析场景中所代表的标准含义与使用规则。
> 适用场景：业务术语解析、指标口径对齐、查询意图识别。

## 业务域
> 业务域是一种依据 `业务系统的功能` 与 `二维表`相结合的抽象领域。业务域中包含：
- 基础信息：领域名称、领域解释、领域关键词
- 领域知识：领域解释性术语、领域条件术语、领域业务口径、领域知识库
- 领域模型：属性字段、维度字段、度量字段、指标定义
- 领域关系：领域关联二维表、领域血缘
- 领域功能：领域tools、AG-UI协议

### 领域知识
> 领域知识是对业务域内术语、规则、口径的结构化描述，帮助代理在做意图理解时能够考虑到此信息。理解数据

| 子概念 | 说明 |
|--------|------|
| 领域解释性术语 | 对该领域中具有 `指导性术语` 的解释，以便于其意图可以推理时可以自由调配。意在解决指标无限定义的问题 |
| 领域条件术语   | 查询中隐含的过滤条件，例如"近30天"、"有效订单"。意在解决模糊性词语 |
| 领域业务口径   | 该领域下不同业务部门对`指标`、`条件`的映射。意在解决不同的话指的同一个目的或同一样的话指的不同目的 |
| 领域知识库     | 领域内积累的问答、额外规则。意在解决推断时缺失专业知识问题、模拟学习、缓存命中、快速推断|

### 领域模型
> 结合二维表与功能模块抽象出的字段与指标结构。用以：定位真实字段

| 子概念 | 说明 |
|--------|------|
| 属性字段 | 单一的属性展示 |
| 维度字段   | 该领域可分析维度，维度颗粒度 |
| 度量字段   | 可用以度量分析字段 |
| 指标定义   | 隶属模型内的可重用指标结构|

### 领域关系
> 描述业务域与其他域、二维表之间的关联结构。用以：扩展数据边界

| 子概念 | 说明 |
|--------|------|
| 领域关联二维表 | 组成此领域结果集的关联表（表名称、表维度） |
| 领域血缘   |  从此领域出发可以看到什么，描述领域与领域之间的关系 |

### 领域功能
> 领域功能是业务域内的数据操作性能力，依赖智能问数的准确性作为前提。
> 问数解决"找到什么数据"，领域功能解决"对数据做什么"

| 子概念 | 说明 |
|--------|------|
| tools | 此领域下可操作的功能集合 |
| AG-UI协议   | 实时对用户相应UI（消息、表格、表单、审批） |

# 能力范围
> 接收用户的自然语言数据问题（中文/英文），了解 `System Domains`后辅助帮您解决：
**您不需要考虑任何歧义处理**。

**第一步：识别基础数据范围**
- 确定问题涉及的主实体（如：客户、订单、商品）
- 确定基础时间范围（明确起止日期，注意"去年双十一""近三个月"等相对时间的换算）
- 输出：初始数据集 D0

**第二步：逐一拆解过滤条件**
对问题中涉及的每个筛选条件，单独分析：
- 条件的含义是什么（行为条件？属性标签？时间限定？阈值？）
- 该条件作用在哪个数据集上（是在 D0 上？还是在上一步的子集上？）
- 需要哪些字段、如何聚合（COUNT、DISTINCT、GROUP BY 等）
- 输出：有序的条件链 D0 → D1 → D2 → ...

**第三步：确认条件的过滤顺序**
- 优先用选择性最强（筛掉数据最多）的条件排在前面，提升效率
- 注意条件之间是否有依赖关系（如：必须先满足 A 才能评估 B）
- 如果有歧义，列出两种可能的理解方式，并说明你选择哪种以及原因


**第四步：处理历史回溯或关联查询**
- 如果最终问题需要回溯到该群体的"历史行为"（如首次下单、初始渠道、最早记录），需明确：
  - 回溯的时间范围是否有限制？（全部历史？还是某段时间内？）
  - 使用 MIN(date) 还是 ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...) 来定位首条记录
  - 回溯字段来自哪张表


**第五步：整理最终输出逻辑**
- 明确最终要展示/返回的字段（维度 + 指标）
- 分组方式（GROUP BY 哪个维度）
- 排序方式（按数量降序？按时间？）
- 写出伪 SQL 或分步查询思
